# Backup Strategy

## Overview

Backups run daily at **03:00** (± 15 min jitter) as a systemd oneshot service via `restic`.
PostgreSQL databases are dumped to disk first, then all paths are sent to the restic repository.

**Destination:** Backblaze B2 (offsite only — no local restic repo is configured)
**Retention:** 7 daily · 4 weekly · 6 monthly snapshots
**Restore script:** `/opt/backup/restore.sh [b2] [snapshots|list|restore|restore-db]`
**Logs:** `journalctl -u restic-backup.service`

## What Is NOT Backed Up (Intentional)

| Path | Size | Reason |
|------|------|--------|
| `/srv/media` | ~145 GB | Re-downloadable via Sonarr/Radarr/Lidarr |
| `/srv/uploads` | ~67 GB | Upload staging area |
| `/srv/comics` | ~46 GB | Re-importable into Komga from source |
| `/srv/downloads` | ~26 GB | In-progress torrent downloads |
| `/srv/photos/encoded-video` | ~12 GB | Regenerated by Immich automatically |
| `/srv/photos/thumbs` | ~2.3 GB | Regenerated by Immich automatically |
| Prometheus TSDB | ~2.2 GB Docker volume | Metrics history — lost on failure |
| Loki log data | ~107 MB Docker volume | Log history — lost on failure |
| Grafana DB | ~25 MB Docker volume | Dashboards are provisioned from `/opt/monitoring/dashboards/*.json` |

> **Note:** Excluding `encoded-video` and `thumbs` from the restic backup would save ~14 GB per snapshot. Add `--exclude /srv/photos/encoded-video --exclude /srv/photos/thumbs` to the `restic backup` call in `roles/backup/templates/backup.sh.j2`.

## Disaster Recovery: Complete Server Failure

**Prerequisites:** a new Ubuntu server reachable as `kazaddum` in your SSH config, with the B2 credentials at hand (in `vault.yml`).

### Step 1 — Provision the new server

```bash
# From the homelab repo on your workstation
make deploy
```

This installs all services via Ansible. All containers will start empty — do not add data yet.

### Step 2 — Stop all services

```bash
ssh kazaddum
for dir in /opt/*/; do
  [ -f "$dir/docker-compose.yml" ] && (cd "$dir" && docker compose down) 2>/dev/null || true
done
```

### Step 3 — Restore files from B2

```bash
ssh kazaddum sudo bash

# List available snapshots
/opt/backup/restore.sh b2 snapshots

# Restore everything to /  (use the latest snapshot ID from above)
/opt/backup/restore.sh b2 restore latest /
```

This restores all backed-up paths to their original locations.

### Step 4 — Restore databases

**Immich:**
```bash
# Start only the DB container first
cd /opt/immich && docker compose up -d immich-db
sleep 5

# Restore from the dump (included in the restic restore above)
/opt/backup/restore.sh b2 restore-db latest
```

**Paperless:**
```bash
cd /opt/paperless && docker compose up -d paperless-db
sleep 5

# The dump was restored to /opt/paperless/db-dump.sql by Step 3
docker exec -i paperless-db psql -U paperless paperless < /opt/paperless/db-dump.sql
```

### Step 5 — Start all services

```bash
make deploy
```

Re-running the playbook starts all containers with the restored config in place.

### Step 6 — Trigger Immich jobs

Immich needs to re-generate thumbnails and encoded videos for any assets whose derivatives were excluded from the backup:

1. Open `https://immich.lab.delval.eu` → **Administration → Jobs**
2. Run **Generate Thumbnails** (all assets)
3. Run **Video Transcoding** (all assets)

## Per-Service Restore

Any single service can be restored from a specific snapshot without touching other services. This is useful for rolling back after an incident (e.g. a Minecraft server attack, accidental config deletion).

```bash
ssh kazaddum sudo bash

# 1. Find the snapshot you want (look at the Time column)
/opt/backup/restore.sh b2 snapshots

# 2. Optionally verify the files inside that snapshot
/opt/backup/restore.sh b2 list <SNAPSHOT_ID> | grep minecraft

# 3. Restore just that service's data (stops the container automatically)
/opt/backup/restore.sh b2 restore-service /opt/minecraft/data <SNAPSHOT_ID>

# 4. Start the service again when ready
cd /opt/minecraft && docker compose up -d
```

The `restore-service` command:
- Accepts any backed-up path (e.g. `/opt/sonarr/config`, `/srv/photos`, `/opt/homeassistant/config`)
- Defaults to `latest` snapshot if no snapshot ID is given
- Defaults to restoring in-place (`/`) — pass a different target as the third argument to restore to a temp location first
- Automatically stops the service's Docker Compose stack before writing files, if a `docker-compose.yml` is found in the service's `/opt/<name>/` directory
- Leaves the service stopped so you can inspect before restarting

### Minecraft attack rollback example

With 7 daily + 4 weekly + 6 monthly snapshots you can roll back up to ~6 months:

```bash
# List all snapshots with timestamps
/opt/backup/restore.sh b2 snapshots

# Pick the snapshot ID from the day before the attack, e.g. abc12345
/opt/backup/restore.sh b2 restore-service /opt/minecraft/data abc12345

# Inspect the world files before starting
ls /opt/minecraft/data/

# Start the server
cd /opt/minecraft && docker compose up -d
```

## Testing Disaster Recovery

Test quarterly or after significant infrastructure changes.

### Option A — Dry-run restore (non-destructive, 5 min)

Verifies the snapshot is intact and restic can read it without touching any live data.

```bash
ssh kazaddum sudo bash -c "
  source /opt/backup/restic-env-b2
  restic check
  restic snapshots
  restic stats latest
"
```

### Option B — Partial restore to temp dir (30 min)

Restores a single service to a temp directory and verifies the files look correct.

```bash
ssh kazaddum sudo bash

source /opt/backup/restic-env-b2
SNAPSHOT=$(restic snapshots --json | python3 -c "import sys,json; snaps=json.load(sys.stdin); print(snaps[-1]['id'])")

# Restore Authelia config to a temp dir
restic restore "$SNAPSHOT" --target /tmp/restore-test --include /opt/authelia
ls -la /tmp/restore-test/opt/authelia/config/

# Verify LLDAP data
restic restore "$SNAPSHOT" --target /tmp/restore-test --include /opt/lldap
ls -la /tmp/restore-test/opt/lldap/data/

rm -rf /tmp/restore-test
```

### Option C — Full restore to a second server (2–4 hrs)

The only true test. Provision a spare machine or a VPS, point `kazaddum` at it temporarily, and run the full disaster recovery steps above. Verify:

- [ ] All services start without errors
- [ ] Authelia SSO login works
- [ ] Immich photos visible and accessible
- [ ] Paperless documents present
- [ ] Home Assistant loads automations and device registry
- [ ] Minecraft world loads correctly
- [ ] Arr stack history and quality profiles intact

## Monitoring

Backup success and size metrics are exported to Prometheus via node-exporter's textfile collector at `/opt/monitoring/textfile/restic.prom`:

| Metric | Description |
|--------|-------------|
| `restic_backup_last_success_timestamp_seconds` | Unix timestamp of last successful run |
| `restic_backup_total_size_bytes` | Total uncompressed size of latest snapshot |
| `restic_backup_files_total` | Number of files in latest snapshot |
| `restic_backup_path_size_bytes{path="..."}` | Size of each backed-up path and its immediate subdirectories |

An alert fires in Uptime Kuma if the backup has not run in more than 25 hours.
